# 上线阻断清单（Go-Live Blockers）

更新时间：2026-03-21  
目标：面向公网稳定开放、可收款、可回滚。

## P0（不完成禁止上线）

1. Git 仓库与发布流程
- 当前目录不是 git 仓库，必须先初始化并建立基础分支策略（`main` / `staging`）。
- 要求：每次上线有可追溯 commit、tag、变更记录、可回滚版本号。

2. 生产环境密钥与配置隔离
- 生产必须单独配置：`LLM_API_KEY`、`JWT_SECRET`、支付密钥、Webhook 密钥。
- 禁止使用默认 `JWT_SECRET`；长度建议 >= 32 且高熵随机。
- 要求：`dev/staging/prod` 三套配置分离，严禁混用。

3. 支付链路可用（真实通道）
- 目前只具备支付入口能力，必须接入真实收银页与回调校验。
- 要求：`/api/subscription/checkout` 可下单，Webhook 可把订阅状态更新到 `active/past_due/canceled`。
- 要求：支付失败、重复回调、签名错误都有明确处理和日志。

4. 鉴权与权限边界验收
- 匿名可用流程必须完整可跑；下载时必须登录+订阅校验。
- 要求：跨用户文档不可读写；token 过期后可刷新；登出后失效。

5. 生产可观测性
- 启用结构化日志与错误监控（建议 Sentry）。
- 要求：每个请求有 `requestId`，任务链路有 `jobId`，关键接口能查成功率和失败原因。

6. 部署与域名
- 必须具备 HTTPS 站点、后端 API 可达、CORS 白名单正确。
- 要求：可从公网完整走通 “输入->澄清->生成->标注修订->登录->支付->下载”。

7. 法务页面
- 上线前至少补齐：用户协议、隐私政策、退款说明。
- 要求：首页与支付前可访问。

## P1（建议首周内完成）

1. 任务可靠性提升
- 当前为内存队列，进程重启后任务丢失。
- 建议：迁移为 Redis/BullMQ 或数据库任务表，支持恢复与追溯。

2. 风控与限流策略精细化
- 分桶限流已做雏形，建议增加按用户/套餐阈值与异常封禁。

3. 运维标准化
- 增加 `staging` 一键部署、数据库备份、告警阈值（失败率/P95 耗时）。

## P2（增长与转化）

1. 价格页与案例页
- 清晰说明免费额度、订阅权益、常见问题。

2. 漏斗埋点
- 记录：访问->生成->修订->登录->支付->下载 的转化率。

---

## 上线验收门槛（建议）

- 端到端主流程成功率 >= 95%（10 次手工冒烟）。
- 生成失败有降级可用稿，不出现“卡死无结果”。
- 修订阶段 A 成功率 >= 98%，阶段 B（全篇修复）失败不阻断编辑。
- 支付回调处理成功率 100%，重复回调幂等。
